12 Macros are a standard feature on many keyboards and Kaleidoscope-powered ones
13 are no exceptions. Macros are a way to have a single key-press do a whole lot of
14 things under the hood: conventionally, macros play back a key sequence, but with
15 Kaleidoscope, there is much more we can do. Nevertheless, playing back a
16 sequence of events is still the primary use of macros.
17 
18 Playing back a sequence means that when we press a macro key, we can have it
19 play pretty much any sequence. It can type some text for us, or invoke a
20 complicated shortcut - the possibilities are endless!
21 
22 ## Using the plugin
23 
24 To use the plugin, we need to include the header, tell the firmware to `use` the
25 plugin, place macros on the keymap, and create a special handler function
26 (`macroAction`) that will tell the plugin what shall happen when macro keys are
27 pressed. It is best illustrated with an example:
28 
29 ```c++
30 #include <Kaleidoscope.h>
31 #include <Kaleidoscope-Macros.h>
32 
33 // Give a name to the macros!
34 enum {
35  MACRO_MODEL01,
36  MACRO_HELLO,
37  MACRO_SPECIAL,
38 };
39 
40 // Somewhere in the keymap:
41 M(MACRO_MODEL01), M(MACRO_HELLO), M(MACRO_SPECIAL)
42 
43 // later in the Sketch:
44 const macro_t *macroAction(uint8_t macroIndex, uint8_t keyState) {
45  switch (macroIndex) {
46  case MACRO_MODEL01:
47  return MACRODOWN(I(25),
48  D(LeftShift), T(M), U(LeftShift), T(O), T(D), T(E), T(L),
49  T(Spacebar),
50  W(100),
51  T(0), T(1) );
52  case MACRO_HELLO:
53  if (keyToggledOn(keyState)) {
54  return Macros.type(PSTR("Hello "), PSTR("world!"));
55  }
56  break;
57  case MACRO_SPECIAL:
58  if (keyToggledOn(keyState)) {
59  // Do something special
60  }
61  break;
62  }
63  return MACRO_NONE;
64 }
65 
66 void setup() {
67  Kaleidoscope.use(&Macros);
68 
69  Kaleidoscope.setup ();
70 }
71 ```

73 ## Keymap markup
74 
75 ### `M(id)`
76 
77 > Places a macro key on the keymap, with the `id` identifier. Whenever this key
78 > has to be handled, the `macroAction` overrideable function will be called,
79 > with the identifier and key state as arguments.
80 >
81 > It is recommended to give a *name* to macros, by using an `enum`.
82 
83 ## Plugin methods
84 
85 The plugin provides a `Macros` object, with the following methods and properties available:
86 
87 ### `.play(macro)`
88 
89 > Plays back a macro, where a macro is a sequence created with the `MACRO()`
90 > helper discussed below. This method will be used by the plugin to play back
91 > the result of the `macroAction()` method, but is used rarely otherwise.
92 >
93 > The `macro` argument must be a sequence created with the `MACRO()` helper!
94 
95 ### `.type(strings...)`
96 
97 > In cases where we only want to type some strings, it is far more convenient to
98 > use this method: we do not have to use the `MACRO()` helper, but just give
99 > this one a set of strings, and it will type them for us on the keyboard. We
100 > can use as many strings as we want, and all of them will be typed in order.
101 >
102 > Each string is limited to a sequence of printable ASCII characters. No
103 > international symbols, or unicode, or anything like it: just plain ASCII.
104 >
105 > Each of `strings` arguments must also reside in program memory, and the
106 > easiest way to do that is to wrap the string in a `PSTR()` helper. See the
107 > program code at the beginning of this documentation for an example!
108 
109 ### `.row`, `.col`
110 
111 > The `row` and `col` properties describe the physical position a macro was
112 > triggered from if it was triggered by a key. The playback functions
113 > do not use these properties, but they are available, would one want to create
114 > a macro that needs to know which key triggered it.
115 >
116 > When the macro was not triggered by a key the value of these properties are
117 > unspecified.

119 ## Macro helpers
120 
121 Macros need to be able to simulate key down and key up events for any key - even
122 keys that may not be on the keymap otherwise. For this reason and others, we
123 need to define them in a special way, using the `MACRO` helper (or its
124 `MACRODOWN()` variant, see below):
125 
126 ### `MACRO(steps...)`
127 
128 > Defines a macro, that is built up from `steps` (explained below). The plugin
129 > will iterate through the sequence, and re-play the steps in order.
130 >
131 > Note: In older versions of the Macros plugin, the sequence of steps had to end
132 > with a special step called END. This is no longer required. Existing macros
133 > that end with END will still work correctly, but new code should not use END;
134 > usage of END is deprecated.
135 
136 ### `MACRODOWN(steps...)`
137 
138 > The same as the `MACRO()` helper above, but it will create a special sequence,
139 > where the steps are only played back when the triggering key was just pressed.
140 > That is, the macro will not be performed when the key is released, or held, or
141 > not pressed at all.
142 >
143 > Use this over `MACRO()` when you only want to perform an action when the key
144 > actuates, and no action should be taken when it is held, released, or when it
145 > is not pressed at all. For a lot of macros that emit a sequence without any
146 > other side effects, `MACRODOWN()` is usually the better choice.
147 >
148 > Can only be used from the `macroAction()` overrideable method.
149 
150 ## `MACRO` steps
151 
152 Macro steps can be divided into two groups:
153 
154 ### Delays
155 
156 * `I(millis)`: Sets the interval between steps to `millis`. By default, there is
157  no delay between steps, and they are played back as fast as possible. Useful
158  when we want to see the macro being typed, or need to slow it down, to allow
159  the host to process it.
160 * `W(millis)`: Waits for `millis` milliseconds. For dramatic effects.
161 
162 ### Key events
163 
164 Key event steps have two variants: one that prefixes its argument with `Key_`,
165 and one that does not. The latter are the `Dr`, `Ur`, and `Tr` variants. In most
166 cases, one is likely to use normal keys for the steps, so the `D`, `U`, and `T`
167 steps apply the `Key_` prefix. This allows us to write `MACRO(T(X))`
168 instead of `MACRO(Tr(Key_X))` - making the macro definition shorter, and
169 more readable.
170 
171 * `D(key)`, `Dr(key)`: Simulates a key being pressed (pushed down).
172 * `U(key)`, `Ur(key)`: Simulates a key being released (going up).
173 * `T(key)`, `Tr(key)`: Simulates a key being tapped (pressed first, then released).
174 
175 ## Overrideable methods
176 
177 ### `macroAction(macroIndex, keyState)`
178 
179 > The `macroAction` method is the brain of the macro support in Kaleidoscope:
180 > this function tells the plugin what sequence to play when given a macro index
181 > and a key state.
182 >
183 > It should return a macro sequence, or `MACRO_NONE` if nothing is to be played
184 > back.
